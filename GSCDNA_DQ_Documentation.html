AT&T GSCDNA Data Quality Cost Monitoring

End-to-End Framework for DQ Cost Tracking, Anomaly Detection, Budget Alerting & Optimization

Version 1.0 | Snowflake Native

Executive Summary

The GSCDNA Data Quality (DQ) Cost Monitoring Framework provides comprehensive visibility into the costs associated with running Data Metric Functions (DMFs) across AT&T's GSCDNA data assets. It enables proactive cost management through automated anomaly detection, budget alerting, and optimization recommendations.

Key Capabilities

  • Cost Tracking: Daily aggregation of DQ monitoring costs per object from DATA_QUALITY_MONITORING_USAGE_HISTORY
  • Anomaly Detection: Statistical analysis using 7-day rolling mean with 2+ standard deviation threshold
  • Budget Management: Per-object budget thresholds with warning/critical multipliers
  • Email Alerting: HTML-formatted email notifications via Snowflake notification integration
  • Cost Forecasting: 6-month projected costs with ~5% growth modeling
  • Optimization: Weekend savings, variability analysis, and frequency recommendations
  • RBAC: Three-tier role hierarchy (Admin > Operator > Viewer)

Daily aggregated cost with day-over-day change percentage using LAG() window function.

Output Columns:

  • TOTAL_CREDITS — Sum of credits
  • TOTAL_COST_USD — Sum of estimated USD cost
  • OBJECTS_COUNT — Distinct objects with data

Aggregates costs grouped by priority tier, joining with tier definitions for context.

Output Columns:

  • OBJECT_COUNT — Objects in this tier
  • TOTAL_COST_30D_USD — 30-day cost for this tier
  • AVG_DAILY_COST_USD — Average daily cost
  • TOTAL_EXECUTIONS — Total DMF executions
  • COST_SHARE_PCT — This tier's share of total cost
VW_DQ_COST_ANOMALIES — Statistical anomaly detection

Applies rolling 7-day window statistics to identify cost spikes exceeding 2 standard deviations.

Detection Logic:

Rolling 7-Day Mean = AVG(cost) OVER (PARTITION BY object ORDER BY date ROWS 7 PRECEDING TO 1 PRECEDING)
Rolling 7-Day StdDev = STDDEV(cost) OVER (same window)
Deviation Score = (actual_cost - rolling_mean) / rolling_stddev

CRITICAL: deviation_score >= 3.0
WARNING:  deviation_score >= 2.0

Output Columns:

  • OBJECT_NAME, LOG_DATE
  • ACTUAL_COST_USD, EXPECTED_COST_USD
  • DEVIATION_SCORE — Number of standard deviations
  • ANOMALY_SEVERITY — CRITICAL, WARNING, or NORMAL
  • IS_ANOMALY — Boolean flag (TRUE when deviation >= 2)
VW_DQ_COST_VS_QUALITY — Cost-quality correlation

Correlates spending with quality pass rates to identify efficiency opportunities.

Performance Quadrants:

QuadrantCostQualityAction
EFFICIENT≤ Average≥ 85%Maintain current approach
HIGH_COST_HIGH_QUALITY> Average≥ 85%Consider reducing frequency
LOW_COST_LOW_QUALITY≤ Average< 85%Increase monitoring investment
NEEDS_OPTIMIZATION> Average< 85%Redesign monitoring strategy
VW_DQ_COST_OPTIMIZATION — Optimization recommendations

Generates actionable optimization recommendations based on cost patterns, weekend utilization, and quality metrics.

Recommendation Logic:

  • REDUCE_FREQUENCY: Pass rate > 99% AND daily cost > $0.015 — checks are running more than needed
  • INVESTIGATE_VARIABILITY: Coefficient of variation > 0.5 — unstable workload pattern
  • REDUCE_WEEKEND: Weekend cost > 80% of weekday cost — reduce non-business-hours monitoring
  • INCREASE_MONITORING: Pass rate < 70% AND cost < $0.010 — under-investing in quality
  • OPTIMAL: None of the above conditions met

Weekend Savings Formula:

Potential Savings = AVG_WEEKEND_COST × 0.5 × 8 (weekend days per month)

Stored Procedures

Five procedures handle the automated monitoring, alerting, and data refresh workflows.

PROC_CHECK_COST_ANOMALIES

Schedule: Daily at 9:00 AM ET

Purpose: Detects cost anomalies by querying VW_DQ_COST_ANOMALIES for today's data and inserting new anomalies into DQ_COST_ANOMALY_LOG.

Flow:

  1. Query anomalies view for today where IS_ANOMALY = TRUE
  2. Skip duplicates (same date + object already logged)
  3. Insert new anomaly records
  4. Generate alert history entries for new anomalies

PROC_SEND_COST_ALERT_EMAIL

Parameters: P_ALERT_TYPE, P_OBJECT_NAME, P_MESSAGE

Purpose: Sends branded HTML email alerts via Snowflake notification integration.

Flow:

  1. Look up recipients from DQ_EMAIL_NOTIFICATION_CONFIG based on object and alert level
  2. Build HTML email with AT&T-branded header, alert badge, message, and timestamp
  3. Call SYSTEM$SEND_EMAIL via notification integration
  4. Mark alert history record as email_sent = TRUE

PROC_DAILY_COST_SUMMARY

Schedule: Daily at 10:00 AM ET

Purpose: Generates daily cost summary and sends to all INFO-level subscribers.

Output Example:

Daily cost summary: Total $0.1425, 10 objects monitored, 2 anomalies detected

PROC_CHECK_BUDGET_ALERTS

Schedule: Every 4 hours

Purpose: Compares today's actual costs against configured budget thresholds per object.

Alert Levels:

  • WARNING — Cost exceeds daily_budget × warning_multiplier
  • CRITICAL — Cost exceeds daily_budget × critical_multiplier

De-duplication: Won't re-alert for the same object within 4 hours.

PROC_REFRESH_COST_DATA

Schedule: Weekly (Monday 8:00 AM ET)

Purpose: Pulls actual DQ monitoring costs from SNOWFLAKE.ACCOUNT_USAGE.DATA_QUALITY_MONITORING_USAGE_HISTORY and updates the cost log.

Flow:

  1. MERGE into DQ_COST_DAILY_LOG from system usage history (last 7 days)
  2. Update existing records, insert new ones
  3. Generate next month's cost forecast using 5% growth rate
Credit-to-USD Rate: $0.00056 per credit (configurable in the procedure)

  1. Open Snowsight and create a new SQL Worksheet
  2. Paste the contents of ATT_GSCDNA_worksheet.sql
  3. Execute sections in order (Section 1 → 8):
    • Section 1: Database & Schema — requires SYSADMIN
    • Section 2: Roles — requires SECURITYADMIN
    • Section 3: Tables — requires SYSADMIN
    • Section 4: Views — requires SYSADMIN
    • Section 5: Procedures — requires SYSADMIN
    • Section 6: Tasks — requires SYSADMIN + EXECUTE TASK grant
    • Section 7: Notification — requires ACCOUNTADMIN

  1. Navigate to Snowsight → Streamlit → Create Streamlit App
  2. Choose settings:
    • App Name: GSCDNA_DQ_Cost_Monitor
    • Warehouse: CORTEX_WH
    • Database: GSCDNA_DQ
    • Schema: GSCDNA_DATA_GOV
  3. Replace the default code with the contents of streamlit_gscdna_dmf_app.py
  4. Click Run to deploy

Cost Model

Credit-to-USD Conversion

Rate: $0.00056 per Snowflake Credit

This rate represents the marginal cost of DQ monitoring compute. It is configured in PROC_REFRESH_COST_DATA and can be adjusted based on your contract pricing.

ESTIMATED_COST_USD = CREDITS_USED × 0.00056

Anomaly Detection Thresholds

ParameterValueDescription
Window Size7 daysRolling lookback for mean/stddev calculation
WARNING threshold2.0 std deviationsCost is unusually high
CRITICAL threshold3.0 std deviationsCost is severely elevated
ExclusionCurrent day excluded from windowUses ROWS 7 PRECEDING TO 1 PRECEDING

Budget Threshold Logic

WARNING  = actual_cost >= daily_budget × warning_multiplier (default 2.0x)
CRITICAL = actual_cost >= daily_budget × critical_multiplier (default 3.0x)

Example: Object with $10 budget
  - WARNING at $20+ (2× budget)
  - CRITICAL at $30+ (3× budget)

Forecast Model

Simple linear growth projection:

Next Month Credits = Current Month Credits × 1.05  (5% growth)
Next Month Cost = Next Month Credits × $0.00056

FAQ & Troubleshooting

Q: Why do I see no data in the dashboard?

Possible causes:

  • Tables have no sample data — run Section 3 (INSERT statements) of the SQL worksheet
  • Role doesn't have SELECT access — ensure you're using DQ_VIEWER_ROLE or higher
  • Wrong database/schema context — the app uses GSCDNA_DQ.GSCDNA_DATA_GOV prefix
Q: Emails are not being sent
  • Verify the notification integration exists: SHOW INTEGRATIONS LIKE 'GSCDNA_DQ_EMAIL%';
  • Ensure recipient emails are in the ALLOWED_RECIPIENTS list
  • Verify recipients have confirmed their email in Snowflake
  • Check DQ_EMAIL_NOTIFICATION_CONFIG has IS_ACTIVE = TRUE for the relevant config
Q: Tasks are not running
  • Check task state: SHOW TASKS IN SCHEMA GSCDNA_DQ.GSCDNA_DATA_GOV;
  • Resume suspended tasks: ALTER TASK TASK_NAME RESUME;
  • Verify EXECUTE TASK grant: must be granted to the owning role
  • Check warehouse is not suspended when tasks attempt to run
Q: Cost data shows $0 or unrealistic values
  • Sample data uses synthetic costs — run PROC_REFRESH_COST_DATA() to pull real data
  • Ensure DMFs are actually running on your GSCDNA tables
  • Check DATA_QUALITY_MONITORING_USAGE_HISTORY has data for your objects
  • Note: There's a 2-hour latency on ACCOUNT_USAGE views
Q: How do I change the anomaly detection sensitivity?

The anomaly threshold is defined in VW_DQ_COST_ANOMALIES. To adjust:

-- Make it more sensitive (flag at 1.5 std dev):
WHEN ... / ROLLING_7D_STDDEV >= 1.5 THEN TRUE

-- Make it less sensitive (flag at 3.0 std dev):
WHEN ... / ROLLING_7D_STDDEV >= 3.0 THEN TRUE

Recreate the view with CREATE OR REPLACE VIEW after making changes.

Q: How do I add a custom DMF to monitoring?
  1. Register the object in DQ_OBJECTS_MONITORED (if not already)
  2. Add the DMF assignment to DQ_DMF_CONFIG: 
    INSERT INTO DQ_DMF_CONFIG (DMF_KEY, OBJECT_NAME, TAG_NAME, DMF_TYPE, SCHEDULE)
VALUES ('CUSTOM_KEY', 'TABLE_NAME', NULL, 'CUSTOM_SQL', '180 MINUTE');
  3. The cost will be automatically picked up by PROC_REFRESH_COST_DATA on next run
Q: Can I use a different warehouse?

Yes. Replace CORTEX_WH in:

  • Section 2 (role grants): GRANT USAGE ON WAREHOUSE <new_wh> TO ROLE ...
  • Section 6 (tasks): ALTER TASK ... SET WAREHOUSE = <new_wh>;
  • Streamlit app deployment settings
